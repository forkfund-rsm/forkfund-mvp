## Project overview

ForkFund is a two-sided B2B fintech MVP built in Python and Streamlit. It turns
fragmented restaurant data into a standardised, explainable Credit Passport and
connects finance-ready Dutch restaurants with professional lenders.

The MVP is complete and functional. The main application file is `app_new_login.py`.

---

## Architecture

### Entry point

`app_new_login.py` — the main Streamlit app (~2,500 lines). Contains:
- Session-state login and registration (restaurant and lender roles)
- Role-based page routing via `st.session_state["current_page"]`
- All page renderer functions (`render_*`)
- CSS injection (`inject_styles`)
- Sidebar rendering (`render_sidebar`)

### Core modules

`src/data_loader.py`
- Loads the 5 CSV files from `data/`
- Validates row counts, keys, and foreign keys
- Computes all derived metrics at runtime: prime cost ratio, DSCR proxy,
  revenue CV, rent-to-revenue, cash-flow margin, data completeness, etc.
- Returns a single metrics DataFrame used by the scorer

`src/scorer.py`
- 9-dimension rules-based scoring engine (deterministic, no ML)
- Inputs: one row of the metrics DataFrame + population context
- Outputs: score (0-100), grade (A-E), risk label, written driver strings,
  peer percentile within revenue band
- Includes smoke tests for the three demo restaurants

### Data layer

All data is synthetic, generated by `scripts/generate_synthetic_data.py`
with a fixed random seed. Files in `data/`:
- `restaurants.csv` — 80 restaurant profiles
- `monthly_bank.csv` — 12 months of bank data per restaurant (960 rows)
- `monthly_pos.csv` — 12 months of POS data per restaurant (960 rows)
- `accounting.csv` — annual financial summary per restaurant
- `lenders.csv` — 8 synthetic lender profiles

No derived scores or metrics are written back to CSV. Everything is
computed at runtime on each page load.

## User roles and routing

The app has two user roles set at login/registration:

**Restaurant** — pages: My Dashboard, My Credit Passport, My Lender Offers
**Lender** — pages: Lender Dashboard, How it works

Routing is controlled by `st.session_state["current_page"]`. This is the
single source of truth for navigation. Do not use `st.session_state["page_override"]`
— it was replaced by `current_page` in the current version.

The sidebar radio only updates `current_page` when the user actively clicks
a different item (detected via `_last_sidebar_pick` comparison).

| `lender_type` | str | Lender's institution type (from registration) |
| `lender_dnb` | str | Lender's DNB registration number |

## Scoring model

Nine dimensions, weights defined in `WEIGHTS` dict in `scorer.py`:

1. Data completeness (10%)
2. Revenue stability (15%)
3. Cash-flow strength (15%)
4. Debt burden and repayment capacity (15%)
5. Prime cost efficiency (10%)
6. Rent and occupancy pressure (10%)
7. POS demand quality (10%)
8. Seasonality and concentration risk (10%)
9. Business maturity (5%)

Grade bands: A (85-100), B (70-84), C (55-69), D (40-54), E (0-39)

The score is pre-underwriting decision support, not a final credit decision.
Do not change the scoring model without updating `docs/mvp_design.md` first.

## Working conventions

**Before making any change:**
1. Read the relevant section of `docs/mvp_design.md` if it touches scoring,
   data schema, or grade bands
2. Identify which session state keys are affected
3. Check whether `current_page` routing needs updating

**Page navigation:**
- Always set `st.session_state["current_page"] = "Page Name"` then `st.rerun()`
- Never use `page_override` (deprecated)
- Add new page names to the `valid_pages` list in `render_sidebar()`
- Add new page names to the routing block in `main()`

**Styling:**
- All CSS is in `inject_styles()` at the top of `app_new_login.py`
- Use CSS class names prefixed with `ff-` for ForkFund components
- Brand colours: navy `#0D1F3C`, teal `#0B6B56`, gold `#C49A2E`
- Do not use em dashes in any user-facing text strings

**Synthetic data:**
- Do not modify CSV files manually
- Regenerate using `scripts/generate_synthetic_data.py` if needed
- Keep the fixed random seed to maintain reproducible demo profiles

**Testing:**
- Run `python3 src/scorer.py` to verify the scoring engine smoke test
- Run `streamlit run app_new_login.py` and test both demo accounts
- Confirm the full restaurant flow: register -> wizard -> verification -> passport -> offers
- Confirm the full lender flow: register -> dashboard -> filter -> view passport -> submit offer

## What is NOT implemented (out of scope)

- Live PSD2 / open-banking API connection
- Live KvK API
- Persistent database (session state only)
- Real authentication (hardcoded demo credentials)
- Predictive ML default model
- Money movement or loan disbursement
- Production-grade security

## AI orchestration used to build this project

This project was built using Claude (claude.ai chat) for architecture decisions,
code generation, and iterative refinement, and Claude Code for direct file editing.

The development approach was design-doc-first: `docs/mvp_design.md` was written
before any code, and served as the authoritative spec. `CLAUDE.md` summarises
that spec for agent context. Each major component was proposed, approved by the
team, then implemented step by step.

Human decisions retained throughout: feature selection, UX flow, demo restaurant
choice, role design, and all product framing. AI was used to generate and refine
code from those decisions, not to make product decisions independently.
